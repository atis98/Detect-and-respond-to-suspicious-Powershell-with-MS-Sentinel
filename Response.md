# This document shows the response steps taken via MS Sentinel after the attack simulation described in the "Attack" file.
## Steps:
- The analytics rule created an alert and an incident that are visible in the Defender portal: <img width="1615" height="832" alt="4_alert" src="https://github.com/user-attachments/assets/7adfb775-1712-4f2f-9596-90bfb50965be" />
<img width="1547" height="770" alt="4_incident" src="https://github.com/user-attachments/assets/95f32b26-f85a-4bb3-9f3d-70d90bcf78b0" />
- I created an automation rule to react to this incident and the similar ones in the future. When an incident is triggered by the analytics rule, it will be set to high priority, assigned to me as owner and run the playbook which I created in MS Sentinel using a Logic App that I designed by myself (not a pre-built one). Here are the automation rule details: <img width="1851" height="874" alt="5_automation_rule" src="https://github.com/user-attachments/assets/01314bdb-5b07-4909-b850-9458c03154be" />
- As you can see, the last step of the automation rule is to run the playbook. On the next screenshots the details of the playbook are visible (the playbook here is basically the trigger of the logic app). When it´s run, it sends a high priority email to me and runs an antivirus scan on the affected device. The first action is possible via the Office 365 connector and the second is via the Defender for Endpoint connector in MS Sentinel - both had to be enabled for this. Another important step is to grant the Sentinel Automation Contributor role to the logic app, otherwise it cannot be added to the automation rule.<img width="1799" height="765" alt="6_logicapp" src="https://github.com/user-attachments/assets/09ffe207-c08d-46f5-8b36-6969cf3241d1" />
